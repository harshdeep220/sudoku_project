# Android Sudoku Solver Automation — Implementation Plan

Automate solving a Sudoku game on a physical Android device: capture the screen over ADB, detect the grid with OpenCV, recognise digits with a small CNN, solve with backtracking, and tap the answer into the app using the "quick-fill" method (select number → tap all matching cells).

---

## Architecture Overview

```
Android Device (USB)
       ↓  ADB screenshot
device_connector.py  ← connect + capture screen into memory
       ↓
vision.py            ← grayscale / blur / threshold → find grid →
                        perspective warp → slice 81 cells →
                        compute screen coordinates (cells + number buttons)
       ↓
ocr.py               ← for each cell: empty check → CNN digit recognition
                        → output 9×9 numpy array (0 = empty)
       ↓
solver.py             ← optimised backtracking → solved 9×9 array
       ↓
executor.py           ← quick-fill loop: for N in 1..9:
                           tap number button N,
                           tap all empty cells needing N
       ↓
main.py               ← orchestrate all the above
```

### Files

#### [NEW] [requirements.txt](file:///z:/sudoku_project/requirements.txt)
Dependencies: `opencv-python`, `numpy`, `uiautomator2`, `tensorflow-cpu` (for CNN), `Pillow`.

---

#### [NEW] [device_connector.py](file:///z:/sudoku_project/device_connector.py)
- Connect to device via `uiautomator2.connect()`.
- `capture_screen()` → returns a raw screenshot as a NumPy BGR array (no disk I/O).
- `tap(x, y)` → convenience wrapper around `device.click(x, y)`.
- `get_screen_size()` → returns `(width, height)` for coordinate scaling.

---

#### [NEW] [vision.py](file:///z:/sudoku_project/vision.py)
- **Pre-processing**: grayscale → Gaussian blur (kernel 7) → adaptive threshold (blockSize 11, C 2).
- **Grid detection**: find contours → pick largest 4-sided polygon → order corners (TL, TR, BR, BL).
- **Perspective warp**: warp to 450×450 px square.
- **Cell extraction**: split warped image into an 81-cell list (50×50 px each), with margin cropping to avoid grid lines.
- **Screen coordinate mapping**: inverse-map each cell centre back to absolute device screen coordinates using the perspective transform matrix.
- **Number button detection**: find the row of 9 buttons at the bottom of the screen (below the grid); return their centre (X, Y) coordinates. Uses a configurable approach — heuristic row splitting based on known app UI layout, with a fallback contour-based method.

> [!IMPORTANT]
> Number-button coordinate detection is the trickiest part. Different Sudoku apps place these buttons differently. The initial implementation uses a **heuristic** (evenly spaced buttons in bottom 15% of screen). If the target app has a non-standard layout, coordinate offsets may need manual tuning.

---

#### [NEW] [ocr.py](file:///z:/sudoku_project/ocr.py)
- Loads the CNN model from `model/digit_model.keras`.
- For each 50×50 cell: centre-crop → resize to 28×28 → normalise.
- **Empty cell detection**: if the cell's white-pixel ratio is below a threshold (~5%), mark as 0.
- Otherwise predict digit 1–9 using the model.
- Returns a `9×9` NumPy int array.

---

#### [NEW] [train_digit_model.py](file:///z:/sudoku_project/train_digit_model.py)
- Downloads MNIST via `tensorflow.keras.datasets`.
- Filters to digits 1–9 only (Sudoku has no 0).
- Augments with small rotations, shifts, and scale changes.
- Trains a small Conv2D → MaxPool → Conv2D → Dense(128) → Dense(9) model.
- Saves to `model/digit_model.keras`.
- Run once: `python train_digit_model.py`.

---

#### [NEW] [solver.py](file:///z:/sudoku_project/solver.py)
- `solve(board)` — in-place optimised backtracking:
  - Pre-computes row/col/box constraint sets for O(1) validity checks.
  - Chooses the empty cell with the fewest candidates (MRV heuristic).
- `is_valid_board(board)` — validates initial board has no conflicts.
- Returns `True`/`False`; board is modified in place.

---

#### [NEW] [executor.py](file:///z:/sudoku_project/executor.py)
- `execute_solution(device, initial_board, solved_board, cell_coords, button_coords, tap_delay=0.05)`
- Compares `initial_board` vs `solved_board` to find cells that were originally empty.
- For each number N from 1 to 9:
  1. Tap the N-th number button.
  2. Collect all cells where `initial_board == 0` and `solved_board == N`.
  3. Tap each of those cells in rapid succession (configurable delay between taps).
- Uses small inter-tap delays to avoid overwhelming the device.

---

#### [NEW] [main.py](file:///z:/sudoku_project/main.py)
- Orchestrator script: `python main.py`.
- Steps:
  1. Connect to device.
  2. Capture screen.
  3. Detect grid + extract coordinates.
  4. OCR → build `initial_board`.
  5. Solve → `solved_board`.
  6. Execute quick-fill automation.
- Prints status at each step (timing, board states, etc.).
- Exits cleanly with a summary.
